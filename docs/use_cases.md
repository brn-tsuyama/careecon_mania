## UC1: バグ調査・要件定義

### 現在のフロー

```
顧客からバグ報告
  → ユーザーがClaudeを対象機能へ誘導（← ここが手作業）
  → Claudeが再現確認・検証（← 手順も説明が必要）
  → Claudeが要件定義として整理  ← ここはすでにできている
  → 開発者へ引き渡し
```

### 地図があれば

```
顧客からバグ報告（テキスト）
  → Claudeが機能マップから対象機能を特定・自律ナビゲート
  → Claudeが期待動作を知っているので「これはおかしい」と自己判断
  → FE+BEの知識ツリーで根本原因まで特定
  → 要件定義として整理 → 開発者へ引き渡し
```

### バグ調査の実例

**対象機能**: 施工管理 > 工程表 > 印刷  
**バグ**: 小工程が存在しない大工程が印刷に表示されない

**UIのみの観察**:
- 印刷プレビューに大工程が表示されない

**BEコードを参照した根本原因**:

```ruby
# PublishedLargeProcess::SearchService#search_by_date
def search_by_date
  @resource = resource.eager_load(:published_small_processes)
  # LEFT JOIN後にWHEREを書くと小工程がNULLの大工程が全て落ちる
  @resource = resource.where('published_small_processes.end_date >= ?', params[:start_date])
end
```

`eager_load`（LEFT JOIN）した後に `published_small_processes.end_date` にWHEREをかけることで、小工程を持たない大工程（NULL行）が全てフィルタされてしまう。SQLレベルのバグ。

**生成された要件定義**: [工程表印刷の要件定義](./requirements/schedule_print_improvements.md)（実例）

## UC2: 新規機能開発

### 地図があれば実現できること

「工程表に工数入力機能を追加する」という依頼に対し、Claude が知識ツリーを参照して以下を生成できる：

```
1. BE: DBマイグレーション（既存テーブルへのカラム追加）
2. BE: Modelの変更（バリデーション・リレーション）
3. BE: GraphQL Typeへのフィールド追加
4. BE: Mutation（既存のupdate系mutationパターンを踏襲）
5. FE: GraphQLクエリファイルの更新
6. FE: 入力コンポーネント（既存のAtomic Designパターンを踏襲）
7. FE: ページコンポーネントへの組み込み
```

**なぜ既存パターンを踏襲できるか**:  
知識ツリーに「このプロジェクトでのGraphQL Mutationの書き方」「Vueコンポーネントの命名規則」「Serviceクラスの責務分担」が入っているため。

## UC3: プレゼン資料・デモデータ

### 必要な知識

- **機能マップ**: どの機能がどこにあるか（ナビゲーション経路）
- **業務コンテキスト**: 各機能がビジネス上何を意味するか
- **手順マップ**: 「デモデータ登録」のような典型的なストーリー

### 地図があれば

- 「新機能Xのリリース資料を作って」→ 機能マップからXに関係する画面を特定 → 自律的にキャプチャ・説明文を生成
- 「デモ用のプロジェクトデータを登録して」→ 手順マップから必要なフォーム入力を順に実行

## UC4: ヘルプページの自動生成

### 現在の課題

- リリースのたびに手書きで更新 → 漏れが発生する
- 画面の変更に追従できない
- 書き手が機能の詳細を理解している前提が必要

### knowledge があれば

```
knowledge/domains/schedule/print.md
  → 印刷機能の概要・操作手順・注意事項がすでに入っている
  → ヘルプ記事のフォーマットに書き出すだけ
```

### 実現できること

- 「印刷の使い方」ページを knowledge から自動生成
- UI が変わったとき（AOM 再キャプチャ → knowledge 更新 → ヘルプ再生成）で追従
- 人間は `[Human]` セクションにある「背景・意図・例外」を書くだけ

### 出力フォーマット例

- Notion / Confluence 形式のヘルプ記事
- Intercom Articles
- 社内 Slack への変更通知サマリー

## UC5: DAP（デジタルアダプション）自動生成

### DAP とは

WalkMe・Pendo・Intercom Product Tours のような「画面上に重ねるガイド」。  
新機能リリース時にウォークスルーやツールチップを表示してユーザーを誘導する。

### 現在の課題

- ウォークスルーのステップを手動で設定する（要素セレクタを手で書く）
- 機能追加のたびにガイドの更新が必要
- 作る工数がかかるので後回しになりがち

### AOM があれば自動生成できる

Playwright の ariaSnapshot が取れる意味的なラベルが、そのまま DAP の素材になる。

```yaml
# captures/aom/schedule-print.yaml（AOM スナップショット）
- dialog "印刷設定":
  - checkbox "基礎工事"
  - radio "小工程の表示 > 表示する"
  - button "印刷する"
```

↓ Claude が変換

```json
// Pendo / WalkMe 向けウォークスルー定義（自動生成）
[
  { "step": 1, "target": "[aria-label='印刷設定']", "text": "印刷設定を開きます" },
  { "step": 2, "target": "[aria-label='基礎工事']", "text": "表示したい大工程にチェックを入れてください" },
  { "step": 3, "target": "[aria-label='印刷する']", "text": "設定完了後、印刷するを押してください" }
]
```

### 実現できること

- 新機能リリース時に DAP ガイドを自動生成して PR に含める
- UI 変更時にウォークスルーを自動更新
- オンボーディングフローの自動生成（画面遷移マップ × 操作手順）

## UC6: MCP公開（将来ロードマップ）

### 現在と将来の関係

```
現在: Claude が MCP 経由でブラウザを「操作する側」
将来: careecon 自体が MCP サーバーになって「操作される側」
```

### なぜ今の取り組みが布石になるか

- 今構築する知識グラフ = MCP サーバーのスキーマ＆コンテキストになる
- 「この操作をすると何が起きるか」の知識 = MCP のツール定義になる
- GraphQL スキーマが整備されていれば MCP サーバーへの変換コストが低い

### 想定される利用形態

外部のAIシステム（他社のClaudeエージェント等）が careecon の MCP サーバーを呼び出して：
- 工程表データの取得・更新
- 日報の自動作成
- プロジェクト進捗の自動レポート

**→ Phase 3 として明確にロードマップに記載。現時点での実装対象外。**
