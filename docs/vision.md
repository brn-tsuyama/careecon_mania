# Careecon Mania — Vision

## なぜこれを作るのか

建設業は慢性的な人手不足だ。  
careecon work のチームも同じ問題を抱えている。

少ないチームで、増え続ける顧客の CS 対応・バグ対応・新機能開発を回す。  
「やるべきことはわかっている。人がいない。」

**これは弱さではない。これが Founder Problem Fit だ。**

自分たちが解きたい問題を、自分たちが一番切実に感じている。  
だからこそ、正しいものが作れる。

---

## どこへ向かうのか

目指すのは二つのゴールが重なる世界だ。

```
【内向き】AI BPO
  careecon チームの CS・QA・開発を AI が担う
  人間は「判断」だけをする

【外向き】MCP-native SaaS
  careecon 自体が AI エージェントに使われる SaaS になる
  顧客の AI が careecon を道具として使う
```

これは別々の話ではない。**同じ基盤の上に立っている。**

内側で AI に仕事をさせるために作る「地図」が、  
そのまま外側に careecon を MCP サーバーとして公開するときのスキーマになる。

今作るものは無駄にならない。

---

## すでに証明されていること

MCP 経由で Claude に careecon の UI を操作させた。  
バグを検証させ、要件定義を生成させた。

**できた。**

ただし、毎回「ストーリー」を渡さないと動けなかった。  
Claude はカーナビなしで運転していた。

**足りなかったのは「地図」だ。**

## 今の資産

careecon work のコードは、知識化に向いている。

### HTML 命名が業務用語そのまま

```yaml
# DOM 走査（MCP）で得られる AOM が、そのままビジネス仕様になる

- button "設定を保存"         ← "btn-submit" ではない
- checkbox "大工程を選択"      ← "item-checkbox" ではない
- textbox "注意事項"           ← "field-01" ではない
```

DOM をスキャンするだけで、画面仕様が日本語で出てくる。  
これは他の SaaS では当たり前ではない。

### Service 層にビジネスロジックが集約されている

```
REST Controller  ──→┐
                     ├──→ Service ──→ Model ──→ DB
GraphQL Resolver ──→┘
```

REST と GraphQL、どちらのパスを辿っても Service に行き着く。  
ビジネスロジックの抽出先が一か所に決まっている。  
ここを読めば、システムが何をしているかがわかる。

### 実例：バグの根本原因がコードに書いてある

```ruby
# PublishedLargeProcess::SearchService#search_by_date
def search_by_date
  @resource = resource.eager_load(:published_small_processes)
  @resource = resource.where('published_small_processes.end_date >= ?', params[:start_date])
  # ↑ LEFT JOIN + WHERE = 小工程のない大工程が全て消える
end
```

顧客が「大工程が印刷されない」と言ったとき、  
地図があれば Claude はこの行に即座に辿り着ける。

## どうナレッジを溜めるか

**Claude がたたき台を作り、人間が育てる。**

```
【コードスキャン（自動）】
  Resolver → Service → Model を静的解析
  DOM 走査で画面・操作を記録
        ↓
【Claude がドラフトを生成】
  knowledge/domains/schedule/print.md
  [Claude生成] セクション: ビジネスロジック・データフロー・既知の挙動
        ↓
【人間がレビュー・追記（最小限）】
  [Human] セクション: なぜ・経緯・例外・顧客の声
  status: draft → reviewed → approved
        ↓
【Claude が次のタスクで参照】
  承認済みナレッジを優先使用 → 精度向上
        ↓
【コード変更を検知して自動更新】
  git diff → 該当ナレッジを再生成 → status: draft に戻す
        ↓ （ループ）
```

人間がゼロから書く必要はない。  
**人間がやるのは「Claudeが読めないもの」だけ。**

| Claude が書く | 人間が書く |
|---|---|
| ビジネスロジックの初稿 | なぜそう設計したか |
| 画面・操作の記録 | 特定顧客の要望で入った経緯 |
| データフロー | 将来変えたい意図 |
| コード変更の追従 | 顧客が実際にどう使っているか |

## どう活用するか

### UC1: バグ調査・要件定義（今すぐ）

```
顧客から報告
  → Claude が機能マップから対象を特定
  → 知識ツリーを辿って根本原因まで特定
  → 要件定義ドキュメントを生成
  → 開発者へ引き渡し
```

人間がやること: 最終確認と承認だけ。

### UC2: 新機能開発（Phase 2）

```
「工程表に工数入力を追加したい」
  → Claude が既存パターンを参照
  → Migration / Model / Service / Controller / Vue コンポーネント を生成
  → 人間がレビューしてマージ
```

知識がある Claude と、ない Claude では、生成コードの品質が根本的に違う。  
既存の命名規則・設計パターン・ビジネス用語に沿ったコードが出てくる。

### UC3: AI BPO（Phase 1〜2 の積み上げ）

```
CS チームへの問い合わせ
  → Claude が知識ベースを参照して即回答
  → 「パスワード変更は設定 > アカウント > セキュリティから」
  → 「その症状は既知バグです。次リリースで修正予定です」

リリース資料・デモデータ登録
  → ナビゲーションマップを持つ Claude が自律的に実行
```

### UC4: MCP-native SaaS（Phase 3）

```
顧客の AI エージェントが careecon を使う時代

「今月の工程表を PDF で出して」
  → 顧客 AI が careecon MCP サーバーを呼び出す
  → データ取得・PDF 生成・返却

「来週の工程を 1 週間延期して」
  → 顧客 AI が工程表の Mutation を実行
```

今作っている知識グラフが MCP のスキーマになる。  
Service 層のビジネスロジックが MCP のツール定義になる。  
**今やっていることが、そのまま次の製品になる。**

## ロードマップ

| Phase | 何をするか | 何が変わるか |
|---|---|---|
| **0** | 両リポジトリの静的スキャン＋知識ベース構築 | Claude が地図を持つ |
| **1** | バグ調査・要件定義の自動化 | CS・QA コストが下がる |
| **2** | 新機能開発への応用 | 開発速度が上がる |
| **3** | careecon を MCP サーバーとして公開 | 顧客の AI が careecon を使う |

Phase 0 が全ての土台だ。ここを丁寧に作る。

人員が少ない今、AI に仕事をさせるために作っているものが、  
そのまま顧客に届ける次のプロダクトになる。

建設業の人手不足を解決するためのツールを作っているチームが、  
**自分たちの人手不足を AI で解決しながら作っている。**

これが Careecon Mania だ。
