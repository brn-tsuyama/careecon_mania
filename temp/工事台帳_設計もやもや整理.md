## ★ コアコンセプト: データのタッチポイント

工事台帳は「データ入力画面」ではなく **「データ集約ビュー」**。

```
工程表（EVM・原価）  ──┐
CRM（売上・請求書）  ──┤→  工事台帳  ←── ここで初めて粗利が見える
出面管理（労務費）   ──┘        ↑
残業計算            ────────────┘
```

各システムが自動でデータを送り込んでくる。
**ユーザーが「タッチ（触る）」のは、自動データが実態と合わないときだけ。**

| ユーザーが触るタイミング | 操作 |
|---|---|
| 自動計算が業務に合わない | Lock & Override（手動上書き） |
| ソースシステムにまだデータがない | OCR or 手動入力（OCRは keiei_plus 既存資産を利用）|
| 数字の根拠を確認したい | ホバーで内訳表示 → ソースへリンク |

**「どこをいじれば数字が変わるか」がわかること = 透明性の本質。**
工事台帳は「答え」を見せるが基本。「入力」は最低限に。パッシブな客にはデフォルト値。

## 前提・設計姿勢

- **CanBe優先**: いきなりTo-Beは顧客業務からの乖離が激しい。まず「原価見積を簡単に」から。
- **市場実験型**: 設計がぶれる箇所はシンプルな設計・実装を選ぶ。後で直す。
- **デジタル弱者想定**: ユーザーはSystem1（反射モード）で動く。自動計算の根拠はホバーで出す。
- **将来**: 見込み原価算出の自動化（TO-BE）・各社のQCDSE算出はMVP後。

## もやもや① 「実際」列の透明性（確定）

### 費目別データソース（確定版）

keiei_plus の `account_items`: 外注費(1) / 材料費(2) / 経費(3) / 労務費(4=未登録) / その他(5)

※ 「実際」は「出来高原価（EV理論値）」と「累積請求原価（AC実額）」の2列に分離（→ もやもや⑪）。

| 費目 | 原価「見積」 | 原価「出来高原価（EV）」 | 原価「累積請求原価（AC）」| 売上「見積」 | 売上「実際」 |
|---|---|---|---|---|---|
| 労務費 | `baseline_cost[p]`（着工時凍結） | `baseline_cost[p] × actual_pct[p][m]` → もやもや③ UC-1 | 未定義（労務費に請求書なし）→ 要設計 | − | − |
| 外注費 | keiei_plus `estimated_cost`（id=1） | `estimated_cost × actual_pct`（出来高按分） → もやもや⑧ | keiei_plus `cost`（id=1）累計 | keiei_plus `order_value` 合計 | keiei_plus `revenue` 合計 |
| 材料費 | keiei_plus `estimated_cost`（id=2） | `estimated_cost`（納入実績日に計上・デフォルトOFF） → もやもや⑧ | keiei_plus `cost`（id=2）累計 | ↑ 同上 | ↑ 同上 |
| 経費 | keiei_plus `estimated_cost`（id=3） | `estimated_cost`（納入実績日に計上） | keiei_plus `cost`（id=3）累計 | | |
| その他 | keiei_plus `estimated_cost`（id=5） | `estimated_cost`（納入実績日に計上） | keiei_plus `cost`（id=5）累計 | | |
| **合計** | | | | keiei_plus `order_value` Documents 合計 | keiei_plus `revenue` Documents 合計 |

**接続キー**: `work_project_id`（両システム共通）
**`cost_price` フィールド追加は不要。** keiei_plus の Documents がそのまま原価明細になる。

### 実際原価の「暫定 vs 確定」

```
請求書未着 → keiei_plus に cost Document なし
  → 費目別フォールバック（→ もやもや③ UC-1「フォールバック」参照）で暫定表示
     ・労務費: baseline_cost × actual_pct（進捗%ベース）
     ・外注費: estimated_cost × actual_pct（出来高按分）
     ・材料費: delivery_actual確定時に estimated_cost を暫定計上

請求書到着 → keiei_plus に cost Document 登録済み
  → keiei_plus の cost Document を優先表示（自動上書き）
```
これも Lock & Override パターンで管理。

**バッジは付けない（確定）**: 暫定/確定の状態を行ごとにバッジ表示するとビジーになる。
ユーザーは数字が欲しいだけ。請求書未着の件数は月次確定ワークフローで「未着 X 件」としてまとめて通知する。

### 透明性の実装方針（デジタル弱者対応）

**ルール: バッジはビジーになるため原則不使用。数字の根拠はホバーで見せる。**

**ホバーで計算内訳を表示（デジタル弱者対応）:**

```
「労務費 原価 ¥960,000」にホバー →
┌────────────────────────────────────┐
```

#### 変数定義

| 変数 | 意味 | データソース |
|---|---|---|
| `p` | 工程（large_process） | careecon_work |
| `u` | ユーザー（工程にアサイン済み） | process_users |
| `m` | 対象月 | — |
| `plan_pct[p][m]` | 工程pの月末時点の**計画**進捗% | 工程表の原計画（baseline） |
| `actual_pct[p][m]` | 工程pの月末時点の**実績**進捗% | `large_processes.progress`（手動入力） |
| `unit_price[u]` | ユーザーuの日当（月給 ÷ 稼働日数） | `project_users.unit_price` |
| `days[p][u]` | 工程pにおけるuの計画人工数 | process期間内の稼働日数（工程の start〜end から導出） |
| `baseline_cost[p]` | 工程pのベースライン労務費（**プロジェクト開始時に凍結**） | `process_baselines`（新規テーブル） |
| `estimated_cost[d]` | 資材・外注dの発注金額 | keiei_plus `estimated_cost` Document |
| `actual_cost[d]` | 資材・外注dの請求書金額 | keiei_plus `cost` Document |
| `delivery_planned[d]` | 資材dの納入予定日 | `documents.delivery_date_scheduled` |
| `delivery_actual[d]` | 資材dの納入実績日 | `documents.delivery_date_actual` |

#### 3指標の定義

```
PV (Planned Value)    = 計画通りに進んでいたら今いくら使っているはず
EV (Earned Value)     = 実際に完了した仕事の当初計画コスト
AC (Actual Cost)      = 実際に使った（発生した）コスト

SV = EV - PV   スケジュール差異（マイナス = 遅延）
CV = EV - AC   コスト差異（マイナス = 予算超過）
```

---

#### 費目別 計算式

**労務費**

```
baseline_cost[p] = Σ(u in p_at_start) unit_price[u] × days[p][u]   ← プロジェクト開始時に凍結

PV[p][m] = baseline_cost[p] × plan_pct[p][m]
EV[p][m] = baseline_cost[p] × actual_pct[p][m]
AC[p][m] = Σ(u in p_current) unit_price[u] × days[p][u] × actual_pct[p][m]
```

**労務費 CV の検知方法（確定）**
- 人員変更がなければ EV = AC → CV ≈ 0（設計上の制約として許容）
- CV ずれの主因は「工程未アサインの出勤」→ **アラート①** で検知・是正
  - 工程に登録されていない人が出勤 → モーダルで工程アサインを促す → AC に反映される
  - これが唯一の現実的な CV 管理手段
- 主指標は **SV**（進捗遅延の検知）。CV はアラート①で防ぐ性格のもの

⚠️ **既知の制約②: days[p][u] は工程期間から導出（人ごとの日数は保持しない）**
- 全アサイン者が工程期間フルで稼働する前提
- 特定の人だけ途中参加・途中離脱するケースは精度が落ちる（MVP許容）

**EV のデフォルト対象（確定）**

```
EV デフォルト = 労務費 EV + 外注費 EV のみ
資機材 EV    = チェックボックスで追加（デフォルト OFF）
```

理由: 資機材は高額になりやすく、グラフに含めると進捗の視認性が著しく落ちる。

---

**材料費**（納入日ベース・ステップ関数）※ EV はデフォルト OFF

```
PV[d][m] = estimated_cost[d]   if delivery_planned[d] が月m内
           0                   otherwise

EV[d][m] = estimated_cost[d]   if delivery_actual[d] が月m内  ← チェックON時のみ集計
           0                   otherwise

AC[d][m] = actual_cost[d]      if delivery_actual[d] が月m内（請求書到着済み）
           estimated_cost[d]   if delivery_actual[d] が月m内かつ請求書未着（暫定計上）
           0                   otherwise
```

**外注費**（出来高按分・確定）

**方式: 出来高ベース（PL方式）に統一（確定）**

日本建設業の慣行として出来高払いが主流 → EV/AC ともに進捗%ベースで算出。請求書はあくまで確定値への差し替えトリガー。

```
PV[d][m] = estimated_cost[d] × plan_pct[p][m]      ← p = 外注dが紐づく工程

EV[d][m] = estimated_cost[d] × actual_pct[p][m]

AC[d][m] = actual_cost[d]    × actual_pct[p][m]    ← 請求書到着後（確定）
           estimated_cost[d] × actual_pct[p][m]    ← 請求書未着（暫定・自動上書き待ち）
```

- CV = (estimated - actual_invoice) × 進捗% → 発注額と請求額の差異のみが CV に現れる
- 請求書到着前は CV ≈ 0（暫定値 = 見積で計算しているため）
- タイミングずれ問題が解消される

※ 外注dが特定工程に紐づいていない場合はプロジェクト全体の進捗%で按分（MVP フォールバック）。

#### 月次配賦（累積→増分変換）

台帳グラフは「当月発生分」で表示するため、累積進捗%から月次増分を計算：

```
monthly_pct[p][m] = actual_pct[p][m] - actual_pct[p][m-1]

EV 月次増分[p][m] = baseline_cost[p] × monthly_pct[p][m]
AC 月次増分[p][m] = Σ(u in p) unit_price[u] × days[p][u] × monthly_pct[p][m]
```

#### フォールバック（データ未入力でも動かす）

| データ状況 | フォールバック | 備考 |
|---|---|---|
| 工程表なし | 工期を日数で均等割りして `plan_pct` を線形補完 | PV のみ影響。EV は actual_pct 依存なので別 |
| `actual_pct` 未入力 | 0（未着手扱い）。EV/AC = 0 | 正しい挙動。未記録 = ゼロ |
| `unit_price[u]` 未設定 | 0（その人は計算対象外） | 人は表示されるが金額ゼロ |
| `delivery_actual` 未設定かつ `delivery_planned` 未来 | EV = 0（まだ納入されていない） | ✅ 修正済み |
| `delivery_actual` 未設定かつ `delivery_planned` 過去 | EV = estimated_cost（計画日に納入とみなす） | 過去の予定日を超えているなら納入済みと推定 |

全材料が工事初日に納入済みとなり EV が大幅に過大になる誤りがあった。納入予定日との比較で判定する方式に変更。

数字が出ないより概算でも出すことを優先。精度低下はホバーで「概算」テキストのみ（アラートなし）。

#### ベースライン凍結タイミング

**トリガー**: 案件ステータスが「着工前 → 施工中」に変わった瞬間に1回だけ実行。

**保存する内容（`process_baselines` テーブル）**:
- 各工程の `baseline_cost`（= アサイン済み全員の unit_price × 計画人工数 の合計）
- 各工程の `plan_pct_series`（= 月ごとの計画進捗%を時系列で保存）← 🚨 **実装必須。これがないと PV が計算できない。JSON形式で月→進捗%のマップを保存する**
- 凍結日時

**例外処理**:
- 工程表が未完成のまま着工 → 工期を均等割りして `plan_pct_series` を仮生成
- 後から「ベースライン再生成」は管理者のみ・初回のみ可能

**なぜ凍結が必要か**: EV は「当初計画コストで測る」が前提。途中で工程を追加・変更しても、凍結済みベースラインには影響しない。変更後の見通しは別途「実行予算」で管理。

### UC-2: 出来高払い（外注先への支払い / 元請けへの請求）

**外注の月次支払いは「発注総額 × 出来高増分%」が業界標準**

**重要: 進捗%は「その外注書類が紐づく工程」の進捗%を使う。案件全体の進捗%ではない。**

```
外注書類dの今月支払い額
  = 発注総額[d] × ( 紐づく工程pの今月末進捗% - 紐づく工程pの先月末進捗% )

例:
  外注d: 型枠工事（工程p: 躯体工事に紐づく）
  発注総額: ¥10,000,000
  躯体工事 先月末進捗: 30%
  躯体工事 今月末進捗: 50%
  今月支払い: ¥10,000,000 × (50% - 30%) = ¥2,000,000
```

**計算ロジック:**

1. 外注書類dに紐づく工程pを特定する
2. 工程pの今月末進捗% と先月末進捗% を `process_monthly_snapshots` から取得（なければ0）
3. 月次増分% = 今月末 - 先月末
4. 発注総額[d] = keiei_plus `estimated_cost` Document（発注書未登録なら `large_process.unit_price` で代替）
5. 支払額 = 発注総額 × 月次増分%

**元請けへの請求も同じ構造（工程ごとの進捗%で按分）:**

```
今月の元請け請求額 = 受注総額（order_value Document） × 紐づく工程の出来高増分%
```

`process_monthly_snapshots` テーブルが前月進捗% の保持に必須。

## もやもや④ 残業ロジックの複雑さ（書き出し）

### 現状の実装（BEコード確認済み）

`OvertimeCalculationService` が計算するのは **overtime_minutes のみ**：

```ruby
overtime_minutes = worked_minutes - in_regular_minutes
# in_regular = 所定労働時間帯との交差部分
# 休日なら全時間がOT扱い
```

**割増金額の計算は未実装。**

### 法定割増の種類と複雑さ

| 割増種類 | 法定率 | 適用条件 | 現状 |
|---|---|---|---|
| 時間外割増 | +25% | 1日8h / 週40h超 | 時間は計算あり・金額なし |
| 深夜割増 | +25% | 22:00〜5:00 | 未実装 |
| 休日割増 | +35% | 法定休日 | 部分的（全時間OT化） |
| 月60h超割増 | +25%（中小特例） | 月60h超の時間外分 | 未実装 |
| 深夜+時間外 | +50% | 夜間残業 | 未実装 |

### グレーゾーンの実態

- 協力会社（下請け）→ 日当制なので割増なし（出面 × 日当のみ）
- 自社社員 → 残業代を実態として出していない会社がある（裁量的運用）
- 適用するかしないかを会社ごとに設定できないと運用できない

### ベストプラクティス提案（シンプル優先）

**MVP: 割増計算はデフォルトON・会社設定でOFF**

> ⚠️ 労働基準法に基づき、時間外・深夜・休日労働には割増賃金の支払いが義務付けられています。正確な原価計算のため、以下の設定を有効にすることを推奨します。

```
会社設定 > 労務費計算
  [ON推奨] 時間外割増 (25%) ※法定
  [ON推奨] 深夜割増  (25%) ※法定
  [ON推奨] 休日割増  (35%) ※法定
  [ON推奨] 月60h超割増(25%) ※法定（中小特例あり）

  割増率を手動で変更することも可能（会社ルールに合わせて調整）
```

- ONにした項目のみ自動計算に乗せる
- 割増が加算されている場合はホバーで「時間外割増含む」と内訳表示
- 割増率は各項目で手動入力可（法定率と異なる社内ルールに対応）
- OFFでも overtime_minutes は記録し続ける（後でONにしたとき遡及できる）
- どの設定でも Lock & Override で手動上書き可能

**「ルールに合わない顧客」への対応:**
設定でOFFにしたうえで、当該行を手動上書き。
= システムを使い続けながら、特殊ケースだけ手動で対処。
「全部手動」より「ほとんど自動 + 手動補正」の方がトータルコストが低い。

## アーキテクチャ方針（確定）

### バリューチェーンとしてのデータ方向

```
現場（careecon_work）
  出面・進捗・勤怠 ← ここでデータが生まれる
       ↓ 上流から下流へ一方向に流れる
工事台帳（keiei_plus）
  集約・確認・承認 ← 初めて粗利が見える層
       ↓ 確定したら財務記録に変換
keiei_plus 収支管理
  Document（cost/revenue）として永続化
       ↓
経営判断（全案件の粗利・資金繰り）
```

keiei_plus と careecon_work を分断してはならない理由:
- **ユーザーが異なる**（現場 vs 経営）
- **管理単位が異なる**（工程 vs 案件 vs 全社）
- **データの方向は1つ**（現場 → 台帳 → 財務）

### 段階的移行計画（CanBe優先）

| Phase | 内容 | 価値 |
|---|---|---|
| **Phase 1（今）** | 集約ビューのみ（READ）。両システムのデータを keiei_plus 上で見せる | 一覧性でExcelに勝つ |
| **Phase 2** | 出面実績 → 労務費 Document を自動生成（WRITE 開始）。keiei_plus に今ない「労務費」が出現 | 粗利に労務費が含まれる |
| **Phase 3** | 外注費・材料費も工事台帳経由で登録できるようにする | 書類起点→工程起点の転換 |
| **Phase 4** | keiei_plus 既存の書類登録 UI を工事台帳に統合・旧 UI 廃止 | UX 統一 |

Phase 2 が最も価値が高い。労務費 Document の生成経路が現在存在せず、工事台帳だけがそれをつなげる。

### 移行期の二重入力について

旧 keiei_plus 書類登録 UI と工事台帳の両方が「原価の入口」になる期間がある。
**これは A/B テストとして許容する**（どちらで入れても同じ Document テーブルに反映されるため矛盾しない）。
バージョン管理があるため整合性リスクも低い。

## プロダクト戦略: Excel 打倒

### 仮説

「一覧性がよい」という評価（PdM・少数顧客から確認済み）は、**実際に操作しても同じように感じられるか？**

→ 市場実験で検証する。結果次第で keiei_plus への本格移行を判断。

### Excel が根強い理由

Excel の本質的な強みは「一覧性」ではなく、**数字を自分でコントロールできている感覚**（自分で式を書いた安心感）。

### 工事台帳が勝てる条件

> **自動で出てきた数字を、ユーザーが信頼できるか**

これが崩れると一覧性がよくても Excel に戻る。

| 指標 | Excel 負けパターン | 工事台帳 勝ちパターン |
|---|---|---|
| 数字の信頼 | 「この数字どこから来た？」→ 確認できない | ホバー内訳で即確認できる |
| 修正の手軽さ | 「合わないから全部手入力した」 | Lock & Override で部分修正 |
| 更新の手間 | 「結局毎月自分でまとめた」 | 出面打刻 → 自動反映 |
| 学習コスト | 「操作がわからない」 | System2 UI + ホバー説明 |

**唯一の killer feature: ホバーで計算内訳が出ること。**
「田中さんが12日来て × ¥6,500 = ¥78,000」が1クリックで出るとき、初めてユーザーは「Excel より速い」と感じる。Excel にはできない。

### 初回リリースの最重要リスク

数字に「抜け」があること。account_item_id NULL や work_project_id 未紐付けで「工事台帳と Excel が合わない」が起きると即 Excel 回帰。

**初回の顧客に出す前に、その顧客のデータを手動整備する。** A/B テストを UI の問題ではなくデータ品質の問題で汚染させない。

## もやもや⑤ 出来高（進捗%）の自動計算（確定）

### 現状の問題

`process.progress`（出来高%）は完全手動入力。出面データと連動していない。
顧客の多くはデータを入れておらず、EVM グラフが動かない。

### A（出来高ベース）と B（出面ベース）の違い（参考）

```
A: actual_price       = 実績進捗% × baseline_cost  ← 台帳の唯一の原価計算源泉（確定）
B: actual_labor_price = 出面 × 単価の累計           ← アラート①②のずれ検知専用
```

**ずれの意味（B は A との比較にのみ使う）：**

| ケース | A（台帳値） | B（アラート参照値） | 意味 |
|---|---|---|---|
| 人が多く来たが進みが遅い | 低い | 高い | 非効率。アラート②候補 |
| 少人数で進んだ | 高い | 低い | 生産性高い |
| 有給・病欠 | 変わらない | 下がる | A の方が実態より高め |
| 外注が来た | 進めば上がる | 0（打刻なし） | 外注は出面外 → 外注費で別計上 |

**確定: 出来高（A）を唯一の原価計算源泉とする**
- `actual_labor_price_cache`（B）は台帳列に表示しない。アラート①②の乖離検知にのみ使用
- アラート①（工程未アサイン出勤）で A の正確性を担保することが前提

### 工事台帳の列設計（確定）

| 列 | 値 | 用途 |
|---|---|---|
| 労務費 計画（PV） | `baseline_cost[p] × plan_pct[p][m]` | EVM スケジュール管理 |
| 労務費 出来高（EV/AC） | `baseline_cost[p] × actual_pct[p][m]` | 台帳・粗利計算の単一原価 |
| 差異（PV - EV） | — | 進捗遅延の可視化（SV） |

### 進捗%の自動計算方針（確定）

**Step 1（即時・DB変更なし）**: `progress` が空のとき計画進捗をデフォルト表示
- `planned_progress_percentage_for(process, as_of: today)` をそのまま使う

**Step 2（中期）**: `progress_type: attendance_based` を追加
- `ProcessDailyLaborCost` の (user_id, work_date) の DISTINCT カウントを分子に使う
- 分母 = アサイン人数 × 稼働日数（計画）
- 工程アサイン（ProcessUser）が必要条件

### ProcessUser 自動作成（確定）

工程アサインを手動で行わない顧客が多い → 打刻データが工程に流れない問題の解決策。

**案件にメンバー追加時 → 全工程に自動アサイン**

```ruby
# ProjectUser after_create
def auto_assign_to_active_processes
  project.schedule&.large_processes&.each do |lp|
    ProcessUser.find_or_create_by!(user: user, large_process: lp, small_process: nil)
    lp.small_processes.each do |sp|
      ProcessUser.find_or_create_by!(user: user, large_process: lp, small_process: sp)
    end
  end
end

# LargeProcess after_create
def auto_assign_project_users
  schedule.project.project_users.each do |pu|
    ProcessUser.find_or_create_by!(user: pu.user, large_process: self)
  end
end
```

- `find_or_create_by!` なので手動アサインは上書きしない
- デフォルト全員・全工程アサイン → クレームが来てから細分化する
- 既存の AllocationMath ロジックはそのまま動く

**もやもや⑫との分担:**
- もやもや⑤（after_create）= 社員・ゲスト全員を対象に、メンバー追加・工程追加のたびに随時登録
- もやもや⑫（waiting→working）= ゲストのみを対象に、工事開始時に未登録分を補完
- 両者とも `find_or_create_by` なので重複しても安全。⑤が正規ルート、⑫はフォールバック兼補完。

## 電帳法・7年保存とスイッチングコスト

Notion コアビジョンより。変更管理 = ロックインの戦略的資産。

## 粗利・販売価格・粗利率の連動 UI

Feature 設計から。工事台帳の「調整」列に必要なインタラクション。

```
原価（自動計算） ← ここを基点にする
  ↓ ユーザーが変えると
粗利率 が連動して更新される

逆算パターン:
  「粗利率 25% で売りたい」→ 販売価格が逆算される
  「販売価格 500万円で契約した」→ 粗利率が逆算される
```

**実装上の注意:**
- 3値（原価・粗利率・販売価格）のうち2つが決まれば3つ目は自動計算
- どれをユーザーが「今触っているか」で計算の向きが変わる
- これも Lock & Override パターンで管理（自動 → 手動上書き）

**推奨実装（確定）: 「最後に触ったフィールドが主」方式（controlled component）**

```typescript
type LastEdited = 'cost' | 'price' | 'margin'

onCostChange(v)   → margin = (price - v) / price * 100
onPriceChange(v)  → margin = (v - cost) / v * 100
onMarginChange(v) → price  = cost / (1 - v / 100)
```

DBに保存するのは `ledger_overrides.manual_value`（target_type: 'margin_rate'）のみ。
売上価格 = cost / (1 - margin_rate/100) はいつでも導出できるので保存不要。

**難易度:** ★★★（UXの状態管理が複雑。FE の設計が肝）

## もやもや⑧ 非労務費のEV/AC計上基準（確定）

### 費目ごとのEV認識タイミングの違い

| 費目 | EV（出来高）の認識タイミング | AC（実費）の認識タイミング |
|---|---|---|
| **材料費・経費・その他** | **納入実績日**（EV デフォルト OFF・チェックで追加） | 請求書の `doc_date` |
| **外注費** | **工程の実績進捗% × 発注総額**（出来高按分・PL方式） | **同左（出来高按分）**。請求書到着後に actual_cost に差し替え |
| 労務費 | 工程の実績進捗% × baseline_cost（→ もやもや③ UC-1 参照） | 同左（進捗%ベース）。アラート①で工程アサインを担保 |

### 材料費・経費・その他の納入日ルール（確定）

**EV はデフォルト OFF（チェックボックスで追加）。高額になりやすく進捗グラフが読みにくくなるため。**

```
納入実績日が設定されている → その日に全額EV計上（チェックON時のみ集計）
納入実績日が未設定 + 納入予定日が過去 → 納入予定日に全額EV計上（計画日に納入とみなす）
納入実績日が未設定 + 納入予定日が未来 → EV = 0（まだ納入されていない）
納入実績日も納入予定日も未設定       → EV = 0
納入予定日のみ（実績未設定）         → PVの計画日として使用
```

※ `delivery_actual 未設定 → project.start_date` は廃止。全材料を工事初日扱いにしてEVが過大になる誤り。

### 外注費のEV = 出来高按分（確定）

UC-2（外注月次支払い計算）と同じロジックを EVM にも使う。
**ただし、外注が工程にアサインされているかどうかで精度を切り替える。**

```
外注費EV = 発注総額 × 実績進捗%
外注費EV（月次） = 発注総額 × (今月末進捗% - 先月末進捗%)
```

#### 按分に使う「進捗%」の2段階フォールバック

```
Case A: 外注がアサインされた工程がある（ちゃんとしたユーザー）
  → アサイン工程のみの出来高平均で按分（精度高い）

Case B: 外注が工程にアサインされていない（データ未入力ユーザー）
  → プロジェクト全体の出来高で按分（フォールバック）
```

どちらのケースかはホバーで内訳表示のみ（バッジなし）。

#### MVP の形（確定）

FR-02（協力会社属性）未実装の現状では、**全員 Case B（全工程按分）** で動かす。

```
MVP: work_customer_id に関係なく全工程進捗%で按分
```

#### 将来の協力会社グルーピングを前提にした実装方針（確定）

`contractor_company_id` カラムを **NULL許容で最初から追加しておく**。
NULL = 自社メンバー or 未紐づけ → Case B に自動フォールバック。
値あり = 紐づけ済み外注 → Case A に自動昇格。**移行スクリプト不要。**

```ruby
# careecon_work migration（FR-02実装時に追加）
add_column :project_users, :contractor_company_id, :bigint,
  null: true, comment: 'keiei_plus.work_customer_id。外注所属会社。NULLは自社メンバー'
add_index :project_users, [:project_id, :contractor_company_id]
```

```ruby
# 計算ロジック（最初からフォールバック込みで実装する）
def ev_progress_for_contractor(document, work_project)
  assigned = work_project.project_users
                         .where(contractor_company_id: document.work_customer_id)
                         .joins(:large_process)
  if assigned.exists?
    assigned.average('large_processes.progress').to_f  # Case A: 担当工程のみ
  else
    work_project.overall_progress                      # Case B: 全工程（MVP/未紐づけ）
  end
end
```

`contractor_company_id` に値を入れるだけで Case A に昇格。API・FE の変更不要。

外注費は「工事が進んだ分だけ価値が生まれる」ので出来高按分が業界実態に合う。
月次支払い計算とEVM計算が一本化される副次効果もある。

### BE への影響（migration 必要）

`documents` テーブルに日付フィールドが `doc_date` 1本しかない。納入日の予定/実績を追加する。

## もやもや⑩ スコープ確定: 工事台帳に入出金ログは持たせない

### 発端のもやもや

「売上（元請けから受け取る）と原価（下請け・調達先に払う）の両方を台帳に持つべきか？」

### 解決した考え方

**軸が違う。混ぜると不整合が起きる。**

```
工事台帳の軸 = 案件
入出金管理の軸 = 取引先（1社が複数案件にまたがる）

例: ○○建設が A工事・B工事・C工事に入っている
  → 支払いは1回にまとめて振り込む
  → 工事台帳に支払いログを持たせると案件との1:1対応が崩れる
```

### 確定: 工事台帳のスコープ

**持つもの: 原価の「発生」と粗利の把握**

| 項目 | 工事台帳 | 理由 |
|---|---|---|
| 費目別原価（見積 vs 実績） | ✅ | 工事原価管理の本質 |
| 売上（受注額） | ✅ カードに1行 | 粗利計算に必要 |
| 粗利・粗利率 | ✅ | 工事採算の可視化 |
| 出来高%・EVM | ✅ | 進捗管理 |
| 出来高払い計算結果 | ✅ 計算のみ | 「いくら払うべきか」の算出 |
| 入金ログ（元請けからいつ入金） | ❌ | 取引先軸・keiei_plus既存機能 |
| 支払いログ（下請けにいつ振込） | ❌ | 取引先軸・会計システムの領域 |
| 入金予定日・支払予定日 | ❌ | 同上 |
| キャッシュフロー | ❌ | 全案件横断で見るもの |

### 出来高払い計算の位置づけ

UC-2（外注月次支払い = 発注総額 × 出来高増分%）は工事台帳で計算する。
ただし**計算結果の表示のみ**。実際の振込記録は持たない。

```
月次タブ:
  ○○建設  今月出来高払い試算: ¥2,000,000  [支払い処理へ →]
                                            ↑ keiei_plus既存機能に飛ばす
```

### テーブル列設計（確定）

入出金・発注ステータス関連の列を削除。バッジ廃止（→ もやもや⑦）。
「原価実績」を「出来高原価」「累積請求原価」に分離（→ もやもや⑪）。

```
費目    │  単価              │  見積原価    │  出来高原価  │  累積請求原価  │  差異
材料費  │  ¥3,200/m²        │  ¥3,000,000  │  ¥2,400,000  │   ¥2,200,000  │ +¥800K
外注費  │  一式              │  ¥8,000,000  │  ¥6,400,000  │   ¥5,500,000  │ +¥2.5M
労務費  │  ¥7,400/日（平均） │  ¥5,000,000  │  ¥3,200,000  │   ¥3,200,000  │ +¥1.8M
経費    │  —                │    ¥500,000  │    ¥320,000  │     ¥200,000  │ +¥300K
────────┼────────────────────┼──────────────┼──────────────┼───────────────┤
合計    │                   │ ¥16,500,000  │ ¥12,320,000  │  ¥11,100,000  │ +¥5.4M
```

差異 = 見積原価 − 累積請求原価。売上・粗利は上部カードに分離（テーブルの列ではない）。

### 単価列の設計（確定）

単価は**過去参照・見積作成のユースケース**で重要。列として保持する。
ただし費目によって「単価」の意味が異なる。

| 費目 | 単価の値 | 算出方法 |
|---|---|---|
| 材料費 | ¥xxx/m²、¥xxx/本 など | Document の単価フィールド |
| 外注費 | 一式（契約金額） | Document の合計額 |
| 労務費 | ¥xxx/日（平均） | `actual_labor_price ÷ 出勤延日数` |
| 経費 | — | 単価の概念が馴染まない |

**労務費の平均単価の意味:**
「前回の大工工事は平均¥7,400/日かかった」として次回見積の参考値になる。
複数人・異なる日当をひとつの参考値に集約したもの。
ホバーで「田中¥8,000×12日、山田¥6,500×8日…の平均」の内訳を表示。

**モックから削除するカラム（確定）:**

| カラム | 削除理由 |
|---|---|
| 入金日（予定/実績） | 入出金管理は取引先軸。工事台帳のスコープ外 |
| 発注（発注済/未発注フラグ） | BEに行単位のデータが存在しない。次の改修スコープ |
| 支払先 | 入出金ログと同セット。スコープ外 |
| 納入日（予定/実績） | モック実装済みだが、発注フラグとセットの概念のため保留 ※ |

※ 納入日は「費用発生日」としてEVMに使う可能性があるため、デザイナーMTGで用途を確認してから判断する。

```
明細（個別）   → 編集可（ユーザーが触るのはここだけ）
費目合計       → 読み取り専用（明細の自動集計）
総額           → 読み取り専用（費目合計の自動集計）
```

明細を編集 → 費目合計が自動増減 → 総額が自動増減。総額の直接編集は不可。

---

### 販売価格のライフサイクル

```
見積段階:  手打ち販売価格 − 見積原価（手打ち） = シミュレーション粗利
施工中:    販売価格（確定）− 累積請求原価（途中） = 実績粗利（途中・正常）
完工後:    販売価格（確定）− 累積請求原価（全確定）= 最終損益
```

工事中の「粗利が良く見える」は正常状態。請求書（売上・原価ともに）は後から出す/来るもの。確定率表示は不要。

## もやもや⑫ 外注費 EV の紐づき問題（確定）

### 問題

`estimated_cost[d] × actual_pct[p][m]` の計算には「document d がどの large_process p に紐づくか」が必要。
現状の documents テーブルに `large_process_id` カラムが存在しないため、外注費 EV を正確に計算できない。

加えて、外注（ゲスト）の会社登録がまだできていないため、ProcessUser が存在しないケースがある。

### 解決方針（確定）

**EV 計算: Case A / B フォールバック**

```ruby
# SubcontractorEvService.call(document, project, month)
# 呼び出し側は 1 行。内部で Case A/B を判定する。

def call(document, project, month)
  if document.large_process_id.present?
    # Case A: 工程紐づきあり（将来）→ その工程の actual_pct
    actual_pct = document.large_process.progress_for_month(month)
  else
    # Case B: 紐づきなし（現在はほぼ全件）→ 全工程進捗%の加重平均
    actual_pct = project.overall_progress_for_month(month)
  end
  document.estimated_cost * actual_pct
end
```

`large_process_id` カラムを追加すれば自動的に Case A に移行できる。移行は段階的。

**ProcessUser 未登録: 工事開始時に自動登録**

```ruby
# waiting → working トリガー（baseline凍結と同タイミング）
def auto_register_guests(project)
  guest_users = project.users.where(role: :guest)
  guest_users.each do |user|
    project.large_processes.each do |process|
      ProcessUser.find_or_create_by(large_process: process, user: user) do |pu|
        pu.source = :auto_initialized  # 自動登録フラグ
      end
    end
  end
end
```

- `find_or_create_by` で冪等（重複作成しない）
- `source: :auto_initialized` で実データと区別できる
- 後から実際の工程紐づけが入れば上書き or 補完

**「クレームは無言より良い」思想**: データなしより不正確でも動く方が価値ある。

### ユーザーへの明示（UI）

追加 UI なし。既存デザイン（ホバー + セル背景色）で対応：

```
auto_initialized の外注費セル → 薄紫背景（仮設定と同じ）
ホバー: 「工事開始日に自動登録されました。
        工程に紐づけると精度が上がります [工程表へ →]」
```

## もやもや⑬ 施工管理者 × EVM 表示設定（確定）

### 問題

施工管理者（member）には労務費金額を非表示にする設計が確定済み。
しかし EVM グラフの縦軸は労務費込みの金額 → 施工管理者が見ると労務費が漏れる。

### 解決方針（確定）

**会社設定（company-wide）で EVM 表示モードを切り替え可能にする。**

```ruby
company.evm_display_for_member  # :progress_only | :amount | :hidden
```

| 値 | 表示内容 |
|---|---|
| `:progress_only` | 進捗%のみ（金額軸なし）← 推奨デフォルト |
| `:amount` | 金額表示あり |
| `:hidden` | EVM グラフ非表示 |

API レイヤーで `role == :member && company.evm_display_for_member == :progress_only` のとき金額軸を除外してレスポンスする。

**MVP 方針**: `:progress_only` ハードコードで始める。設定 UI（管理画面）は Phase2。

**既存機能との関係**: 既存の施工管理者権限管理（`project_users` ロール）とは別に、会社設定テーブルへのカラム追加が新設必要。

## もやもや⑭ EV/PV スコープ統一（確定）

### 問題

材料費チェックボックスで EV から材料費を除外できるが、PV（見積）は材料費込みのまま。
EV と PV のスコープが揃わないと SV（スケジュール差異）が嘘になる。

### 解決方針（確定）

材料費チェックの ON/OFF で **EV と PV を同時に**切り替える。

```
材料費 OFF（デフォルト）:
  EV = 労務費EV + 外注費EV
  PV = 労務費PV + 外注費PV
  SV = EV - PV  ← 正確

材料費 ON:
  EV = 全費目EV
  PV = 全費目PV
  SV = EV - PV  ← 正確
```

チェックボックスの切り替えでグラフが再計算される。通知なし（ユーザーが見て判断する）。

**リスク**: 材料費が高額な工事では ON にしたときグラフの縦軸スケールが大きく変わる。
デフォルト OFF にしておくことで通常は回避できる。

## もやもや⑮ 労務費 CV ≈ 0 問題（確定）

### 問題

```
EV = baseline_cost × actual_pct
AC = Σ(u) unit_price[u] × days[p][u] × actual_pct
baseline_cost の定義: Σ(u in p_at_start) unit_price[u] × days[p][u]
→ EV = AC が恒等式。CV = 0 が構造的に確定。
```

CV が意味を持つのはプロジェクト開始後に人員追加・削減があった場合のみ。

### 解決方針（確定）

**CV ≈ 0 のままでよい。**

```
CV 発生条件:
  - 人員追加後: AC > EV → CV < 0（コスト超過シグナル）
  - 人員削減後: AC < EV → CV > 0（コスト節減）
  - 出面ずれ:   アラート①で検知・工程アサインを促す（もやもや⑥）

主指標:
  労務費 → SV（進捗遅延）が有効
  CV     → 人員変動の検知のみに使う
```

「指標としてほぼ機能しない」のは労務費の特性上正しい。無理に CV を活用しようとせず、SV に集中する。
