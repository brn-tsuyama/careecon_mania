# 工事台帳 デザイナーMTG 論点整理
日付: 2026-07-01
対象: 工事台帳（新機能）のUI設計

---

## 前提共有（MTG冒頭で確認）

工事台帳は「データ入力画面」ではなく **「データ集約ビュー」**。

- 自動でデータが集まってくる（工程表・出面・請求書）
- ユーザーが触るのは「自動値が実態と合わないとき」だけ
- **自動で出た数字をユーザーが信頼できるか** = Excel打倒の唯一の条件

ユーザーは建設業の経営者・事務担当（デジタル弱者、System2：熟考モード）。

## 論点② バッジシステム（**廃止確定** → ホバーに集約）

### 決定事項
セルごとにバッジ（🤖/✏️/📄）を表示するとビジーになるため、**バッジは原則不使用**。
数字の根拠はホバー（ツールチップ）でのみ表示する。

### 残す唯一の状態表示
**Lock（手動入力中）のみ**、セルを視覚的に区別する（背景色の変化程度）。
→ 「ここだけ自動計算から外れている」を一目で判別できれば十分。

### デザイナーへの確認
- Lock状態のセルをどう表現するか（背景色・枠線・アイコン最小限どれが最もシンプルか）
- 「仮設定・データ未入力」セルの表現をバッジ以外でどう見せるか（背景薄色 or ハイフン表示 or 何もしない）

## 論点③ Lock & Override UX（案あり・確認）

### 何を決めたいか
自動計算値を手動上書きする → 元に戻す、の操作フロー。

### 現状案（ゴースト値パターン）

**通常状態（自動計算中）**
```
┌─────────────────────────────────────┐
│  労務費 原価                         │
│  ¥960,000  🤖 自動計算             │
│                           [修正する] │
└─────────────────────────────────────┘
```

**「修正する」クリック → 確認モーダル（重い操作なので確認あり）**
```
┌─────────────────────────────────────────┐
│  ⚠️  手動入力に切り替えます              │
│                                         │
│  この値は工程表・出面データから          │
│  自動的に計算されています。             │
│  手動入力に切り替えると、               │
│  以降の出面変更があっても               │
│  この値は自動更新されません。           │
│                                         │
│     [キャンセル]  [手動入力に切り替え]  │
└─────────────────────────────────────────┘
```

**手動入力状態（Lock済み）**
```
┌────────────────────────────────────────┐
│  労務費 原価                            │
│  ┌──────────────────────────────────┐  │
│  │  ¥ [  800,000  ] ✏️ 手動入力    │  │
│  └──────────────────────────────────┘  │
│  計算値: ¥960,000  [↩ 計算値に戻す]   │  ← ゴーストで常に表示
└────────────────────────────────────────┘
```

**「↩ 計算値に戻す」→ 確認なしで即戻す（軽い操作）**
- 修正は重い（モーダルあり）・戻すは軽い（即時）の非対称性が安心感になる

### ゴースト値の更新
- 手動Lock中も計算は続く → ゴーストが変わる
- 差が大きくなったらアラート（オプション）:
  ```
  ⚠️ 計算値（¥1,100,000）と手動値（¥800,000）の差が30万円以上になっています
  ```

### 階層ルール
```
大工程「大工工事」← 子の合計（手動入力ボタンを出さない）
  └── 小工程A    ← ここだけ「修正する」ボタンを出す
  └── 小工程B
```
大工程は「内訳を見る」ボタンで小工程に展開。

### デザイナーへの確認
- ゴースト値の表示位置・文字サイズ（読めるが目立ちすぎない）
- スマホ時の折りたたみ案（後述・論点⑨）

## 論点⑤ データ未入力者への誘導 UX（方針あり・詳細未設計）

### 何を決めたいか
単価や出来高が未入力の顧客に「どこを入れれば台帳が動くか」を教える UI。

### 方針（確定）
- 空欄のままにしない → **仮の値で計算して表示**
- 「仮で計算しています」と明示 → ホバーで「どこを直すか」リンク

### 仮値の設定
| 項目 | 仮のデフォルト値 |
|---|---|
| 労務費単価（未入力の場合） | 月収35万円 ÷ 20日 = ¥17,500/日（会社設定で変更可） |
| 進捗%（未入力の場合） | 工程の計画進捗%（日数ベース） |
| 外注費（請求書未登録の場合） | 工程の原価見積 × 進捗% |

### 誘導ホバー案
```
労務費 ¥420,000  📋 仮設定  にホバー →
┌──────────────────────────────────────────────────┐
│ 📋 仮の単価を使って計算しています                  │
│                                                  │
│ 田中 太郎: 日当 ¥17,500（仮）× 24日 = ¥420,000   │
│                                                  │
│ 正確な値にするには:                               │
│ → [メンバー設定で単価を入力する]                  │
└──────────────────────────────────────────────────┘
```

### デザイナーへの確認
- 「仮設定」行の背景色（薄紫の案）
- ダッシュボードに「仮設定が残っている工事 X 件」バナーを出すか
- 誘導リンクのCTA文言・デザイン

## 論点⑥ 粗利・販売価格・粗利率の連動 UI（方針あり・詳細未設計）

### 何を決めたいか

```

販売価格 変更 → 粗利率が変わる

- ロックがかかった状態（手動入力）との見た目の区別

## 論点⑦ アラート設計（案あり・確認）

### 2種類は性質が異なる

| | 🔴 データ破綻（最優先） | 🟡 赤字リスク |

| 例 | 工程未アサインで出勤 | 予算超過・EV-AC乖離 |
| 問題 | 原価計算できない（¥0が嘘の値） | 赤字になりそう |
| UX | **モーダル・口うるさく** | **どこかに表示しておく** |

### 🔴 アラート①（工程未アサイン）
```
┌─────────────────────────────────────────────────────┐
│ 🔴 労務費を計算できません                            │
│                                                     │
│ 田中 太郎 が 6/28 出勤しましたが、この工程に        │

│                                                     │
│ [ ] 5日間表示しない                                 │
│            [今すぐアサインする]                     │
└─────────────────────────────────────────────────────┘
```

### 🟡 アラート②③（赤字リスク）
```
工事台帳のサイドバーやサマリー欄に小さく:
  🟡 労務費が予算の 80% に達しました（大工工事）[確認する]

## 論点⑧ OCR 入口の2分割（方針あり・確認）

### 何を決めたいか
同じ「OCRで読み取る」でも2種類あり、入口を明確に分ける。

| 種類 | 対象 | 使う人 | デバイス |
|---|---|---|---|
| 請求書 D&D | 外注先からのPDF請求書 | 事務担当 | PC |
| レシート撮影 | 現場の材料費レシート | 現場監督 | スマホ |

同じ画面に両方出すと混乱する → 入口を完全に分ける。

### デザイナーへの確認
- PC版（請求書D&D）のドロップゾーンの配置
- スマホ版（レシート撮影）のカメラ起動UI
- OCR確認ステップ（読み取り結果を確認してから保存）のフロー

## 論点⑨ 残業割増設定 UI（案あり・確認）

### 対象画面
会社設定 > 労務費計算設定

### 前提
- デフォルトはすべてOFF（グレー運用の会社がいきなり計算されないように）
- 会社全体（一括）と社員個別の2階層で設定できる

### 会社設定（一括）

```
┌──────────────────────────────────────────────────────────┐
│  会社設定 > 労務費計算                                     │
│  ⚠️ OFFの場合、残業時間は記録されますが金額に反映されません │
│                                                          │
│  種類          │ 適用  │ 割増率  │ 対象条件              │
│  時間外割増    │ ○ OFF │ 25 %   │ 1日8h超               │
│  深夜割増      │ ○ OFF │ 25 %   │ 22:00〜5:00           │
│  休日割増      │ ○ OFF │ 35 %   │ 法定休日              │
│  月60h超割増   │ ○ OFF │ 25 %   │ 月60h超分             │
│                                              [保存する]   │
└──────────────────────────────────────────────────────────┘
```

ONにすると「この会社の全社員に適用されます。労務費が再計算されます」確認モーダル。

### 社員個別設定
案件 > メンバー > 各メンバーの詳細 > 労務費設定
デフォルト「会社設定に従う」→ 個別設定に切り替えると同じトグル表が展開。

### デザイナーへの確認
- トグルのON/OFFは色＋文字で示す（色覚対応）
- 割増率の%入力：整数のみ、スピナー不要、直接入力
- 社員個別設定の場所（メンバー詳細画面か、工事台帳内か）

## 論点⑪ スマホ対応方針（一部あり・確認）

### 何を決めたいか

| 画面 | PC | スマホ |

```
┌────────────────────────────────┐
│  労務費 原価                    │
│  ¥800,000  ✏️                  │
│  [計算値と比較] [↩ 戻す]       │

```

## MTG で決めたい優先度

| 優先度 | 論点 | 理由 |
|---|---|---|
| 🔴 最優先 | ①全体構造・タブ | ここが決まらないと他が動かない |

| 🔴 最優先 | ③Lock & Override | 実装の基盤 |
| 🟡 重要 | ⑦アラート | 実装順が早い |

| 🟢 後でよい | ④ホバー内訳 | 方向感はある |
| 🟢 後でよい | ⑥粗利連動 | FE設計で解決できる |
| 🟢 後でよい | ⑧OCR入口 | 設計ほぼ固まっている |
| 🟢 後でよい | ⑨残業割増設定 | 会社設定画面。独立している |
| 🟢 後でよい | ⑩出面一覧 | 別画面として後から |
| 🟢 後でよい | ⑪スマホ対応 | PC版が固まってから |
